# Checklist.md — Activity A (i) & (ii)

## Part (i): Research & Preparation

### Research Tasks
- [ ] Investigate how digital solutions are used across industries.
- [ ] Collect factual notes only (no analysis).

### Required Coverage
- [ ] Hardware applications.
- [ ] Software applications.
- [ ] Emerging technologies.
- [ ] Meeting diverse user needs (accessibility, usability).
- [ ] Industry-specific guidelines and regulations.

### Security & Compliance
- [ ] Encryption methods.
- [ ] Secure data storage.
- [ ] Authentication approaches.
- [ ] Data protection laws (GDPR).
- [ ] Accessibility standards (WCAG).

### Research Quality
- [ ] Use credible sources (industry reports, standards bodies, tech blogs).
- [ ] Organise notes under:
  - [ ] Technologies
  - [ ] User Needs
  - [ ] Regulations
- [ ] Keep notes factual and concise.
- [ ] Prepare notes for Appendix A.

## Part (ii): Proposal Development

### 1. Business Context

#### Client Brief Review
- [ ] Summarise client goals and problems.
- [ ] Keep sector-neutral.
- [ ] Link goals to user needs.
- [ ] Mention relevant security/compliance requirements.

#### Stakeholders
- [ ] Identify technical stakeholders.
- [ ] Identify non-technical stakeholders.
- [ ] Identify end users.
- [ ] Include compliance/security stakeholders.

#### Constraints
- [ ] Budget limitations.
- [ ] Time constraints.
- [ ] Resource constraints.
- [ ] Legal/security constraints (GDPR, accessibility).

#### Empathy Mapping
- [ ] Create a simple diagram.
- [ ] Consider privacy concerns.
- [ ] Consider accessibility needs.

#### SWOT Analysis
- [ ] Strengths.
- [ ] Weaknesses (include security risks).
- [ ] Opportunities.
- [ ] Threats (include compliance/cyber risks).

#### User Stories & Acceptance Criteria
- [ ] Write clear user stories.
- [ ] Make acceptance criteria measurable.
- [ ] Include security requirements (encryption, authentication).

---

### 2. Functional & Non‑Functional Requirements

#### Functional Requirements
- [ ] Define system functions.
- [ ] Ensure requirements are measurable.

#### Non‑Functional Requirements
- [ ] Performance.
- [ ] Usability.
- [ ] Security (authentication, encryption).
- [ ] Compliance (GDPR, accessibility).

---

### 3. Decomposition of Requirements
- [ ] Break requirements into smaller tasks.
- [ ] Use tables or diagrams.
- [ ] Show dependencies.
- [ ] Include security/compliance tasks.

---

### 4. KPIs & User Acceptance
- [ ] Define measurable KPIs.
- [ ] Include security KPIs (for example, zero breaches).
- [ ] Align KPIs with compliance standards.
- [ ] Define user acceptance criteria.

---

### 5. Proposed Solution
- [ ] Describe the solution clearly.
- [ ] Include diagrams or visuals.
- [ ] Highlight security measures (encryption, secure APIs).
- [ ] Highlight compliance measures (GDPR, accessibility).

---

### 6. Justification
- [ ] Explain how the solution meets client needs.
- [ ] Explain how user needs are addressed.
- [ ] Explain how risks are mitigated.
- [ ] Reference relevant laws (GDPR, accessibility).
- [ ] Include cyber‑risk mitigation.

---

## Appendices
- [ ] Include all research notes from Activity A (i).
- [ ] Ensure notes are factual only.
- [ ] Reference appendices in the main document.
- [ ] Ensure research supports proposal decisions.
